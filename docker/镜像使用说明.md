# BindCraft 三镜像使用说明

> 面向"拉取已构建镜像并测试"的读者。讲清楚三个镜像各是什么、各自的输入输出、它们如何协作,以及怎么一步步跑起来验证。
> 想看构建/CI/扩容细节,见同目录 `../README-docker.md`(更详尽的开发者参考)。

---

## 1. 这是什么

BindCraft 是一个从头设计蛋白 binder 的流水线(AlphaFold2 反向传播 + ProteinMPNN + PyRosetta 打分)。原版是单 GPU 单进程跑完全部步骤。

本套 Docker 把它**拆成三个独立镜像**,在单机上用 Docker Compose 协同,从而支持**批量输入、批量计算**。三个镜像各装各的依赖栈,互不污染:

| 镜像 tag | 角色 | 装了什么 | 要 GPU? |
|---|---|---|:---:|
| `ai4science/bindcraft:rosetta` | **打分服务**(常驻 HTTP) | PyRosetta + DAlphaBall + dssp(**无 jax**) | 否 |
| `ai4science/bindcraft:gpu` | **设计 worker** | jax + colabdesign + AF2 权重(**无 pyrosetta**) | 是 |
| `ai4science/bindcraft:orchestrator` | **调度/聚合** | 仅 pandas(**无 jax、无 pyrosetta**) | 否 |

拆分靠一个环境变量 `BINDCRAFT_MODE`(`local` / `gpu` / `rosetta`)切换,科学代码本身几乎没改。

---

## 2. 拉取镜像

DockerHub 不通时用镜像站 `docker.1ms.run`:

```bash
docker pull docker.1ms.run/ai4science/bindcraft:rosetta
docker pull docker.1ms.run/ai4science/bindcraft:gpu          # 大(CUDA+jax+5.3GB 权重,约 10GB+),慢
docker pull docker.1ms.run/ai4science/bindcraft:orchestrator
```

> 镜像站对大层偶发 `unknown blob` / `unexpected EOF`。失败就重试,已下载的层会缓存续传:
> ```bash
> for i in $(seq 1 8); do docker pull docker.1ms.run/ai4science/bindcraft:gpu && break; sleep 10; done
> ```

在仓库根目录建 `.env`,让 compose / 测试脚本用这三个 tag:

```ini
GPU_IMAGE=docker.1ms.run/ai4science/bindcraft:gpu
ROSETTA_IMAGE=docker.1ms.run/ai4science/bindcraft:rosetta
ORCH_IMAGE=docker.1ms.run/ai4science/bindcraft:orchestrator
WORKERS=1            # gpu-worker 副本数;单 GPU 必须保持 1(一个复合物可能要 ~32GB 显存)
ROSETTA_REPLICAS=1   # rosetta 服务副本数;多 worker 时可加
CUDA=12.4
```

## 3. 三个镜像各自的输入 / 输出

### 3.1 `rosetta` —— PyRosetta 打分 HTTP 服务

启动后是一个**常驻 HTTP 服务**(默认 `:8000`),把 BindCraft 用到的 5 个 PyRosetta 操作暴露成接口。它不主动干活,等 gpu worker 来调。

- **入口**:`python -u /app/docker/rosetta_service.py`(镜像默认 CMD)
- **输入**:HTTP POST JSON,**只传文件路径**(PDB 本体在共享卷上,不走网络)
- **输出**:JSON `{"ok": bool, "result": ..., "error": str}`

| 接口 | 入参 | 返回 |
|---|---|---|
| `GET /health` | — | `200 "ok"`(healthcheck 用) |
| `POST /pr_relax` | `{pdb_file, relaxed_pdb_path}` | `null`(把 relax 后的 PDB 写到 `relaxed_pdb_path`) |
| `POST /score_interface` | `{pdb_file, binder_chain}` | `{scores, interface_AA, interface_residues}` |
| `POST /unaligned_rmsd` | `{reference_pdb, align_pdb, reference_chain_id, align_chain_id}` | `{rmsd}` |
| `POST /align_pdbs` | 同上 | `null`(写回 `align_pdb`) |

> ⚠️ **必须单线程**:PyRosetta 的 mover(如 FastRelax)在非主线程执行会**段错误**直接崩掉服务。所以服务用单线程 `HTTPServer` + 全局锁串行处理;并发靠**多副本**(`ROSETTA_REPLICAS`)而不是多线程。这是本套镜像里一个已修复的关键点,改副本数即可扩 CPU 吞吐,别试图改成多线程。

### 3.2 `gpu` —— 设计 worker

跑未经改动的 `bindcraft.py` 设计主循环。所有对 Rosetta 的调用经 HTTP 转发到 `rosetta` 服务。

- **入口**:`docker/worker.py`(默认 entrypoint 会先等 rosetta `/health` 通过,最长 300s,再开始认领 job)
- **输入**:
  - 共享卷 `/workspace/queue/pending/` 里的 target JSON(由 orchestrator 投放)
  - AF2 权重已**打包在镜像内** `/app/params`,运行时不下载、不挂卷
- **输出**(全部写到共享卷):
  - `/workspace/designs/<name>/` —— 每个 target 的产物 + 中间 PDB + `final_design_stats.csv` 等
  - job 文件流转到 `queue/done/` 或 `queue/failed/`

> `BINDCRAFT_MODE=gpu` 下,`bindcraft.py` 里的 `pr.init(...)`、`pr_relax`、`score_interface` 等名字被 `functions/rosetta_client.py` 这个同名 shim 接管,真正的 PyRosetta 在 rosetta 容器里跑。gpu 镜像里**没有也不需要** pyrosetta。

### 3.3 `orchestrator` —— 调度 + 聚合

不碰 GPU、不碰 PyRosetta,只做入队和汇总。

- **入口**:`docker/orchestrator.py`
- **输入**:仓库根目录 `batch_inputs/` 下的一个或多个 target JSON
- **行为**:把它们复制到 `/workspace/queue/pending/` 供 worker 认领;worker 跑完后,把每个 target 的 `final_design_stats.csv` 聚合
- **输出**:`/workspace/combined_final_stats.csv`(跨 target 合并结果)

## 4. 它们怎么协作

```
            batch_inputs/*.json
                  │  (orchestrator 复制)
                  ▼
       /workspace/queue/pending/*.json
                  │  (gpu-worker 用 os.rename 原子认领)
                  ▼
   ┌──────────────────────────────────────┐
   │  gpu-worker:  跑 bindcraft.py 设计循环   │
   │     │  需要 relax / score / rmsd / align │
   │     └──HTTP──►  rosetta 服务 (:8000)     │
   │                  │ 读写 PDB              │
   └──────────────────┼──────────────────────┘
                       ▼
        共享卷 /workspace/designs/<name>/   ← PDB 只经这里交接,不走网络
                  │  (orchestrator 聚合)
                  ▼
        /workspace/combined_final_stats.csv
```

关键约束(测试时最容易踩):

1. **三个容器必须把同一个 workspace 卷挂在相同路径 `/workspace`**。gpu 和 rosetta 之间 PDB 是按**路径**传的,路径对不上就 `FileNotFound`。
2. **target 的 `design_path` 必须在 `/workspace` 下**(例:`/workspace/designs/PDL1/`)。
3. 启动顺序:`rosetta`(healthy)→ `gpu-worker` → `orchestrator`。compose 已配好依赖,正常 `docker compose up` 即可。

## 5. 测试步骤(由浅入深)

### Level 1 — 依赖隔离(无 GPU,~10 秒)

确认三个镜像各自只装了该装的栈。注意:这些镜像基于 micromamba,**python 只有走默认 entrypoint 才在 PATH 里**,所以命令直接作为参数传给镜像(别加 `--entrypoint python`):

```bash
docker run --rm docker.1ms.run/ai4science/bindcraft:gpu          python -c "import jax, colabdesign; print('gpu ok, 无需 pyrosetta')"
docker run --rm docker.1ms.run/ai4science/bindcraft:rosetta      python -c "import pyrosetta; print('rosetta ok, 无需 jax')"
docker run --rm docker.1ms.run/ai4science/bindcraft:orchestrator python -c "import pandas; print('orchestrator ok')"
```

### Level 2 — Rosetta RPC 边界(无 GPU,关键,~30 秒) ⭐

这是拆分后最核心的新链路:**GPU 进程 → HTTP → Rosetta 服务,PDB 经共享卷交接**。一条命令端到端验证(起服务 → `/health` → 用内置 `example/PDL1.pdb` 真跑一次 `pr_relax` → 校验产物 → 清理):

```bash
bash docker/smoke_test.sh
```

输出 `summary: N passed, 0 failed` 即通过。**不需要 GPU**,是验证"重构没坏"最快、最有价值的一步。

> 想手动验证 rosetta 服务本身:
> ```bash
> docker run -d --name rtest -p 8000:8000 docker.1ms.run/ai4science/bindcraft:rosetta
> sleep 20 && curl localhost:8000/health      # 期望 ok
> docker rm -f rtest
> ```

### Level 3 — GPU 可见性(需 GPU)

```bash
docker run --rm --gpus all docker.1ms.run/ai4science/bindcraft:gpu python -c "import jax; print(jax.devices())"
# 期望打印含 gpu/cuda 的设备;为空则查宿主 nvidia-container-toolkit
```

### Level 4 — 端到端批量冒烟(需 GPU,较慢)

用内置精简靶标跑通全链路(只求 1 个最终设计、binder 很短):

```bash
mkdir -p batch_inputs
cp docker/examples/PDL1_smoke.json batch_inputs/

# 用无过滤器加快出结果
WORKERS=1 DEFAULT_FILTERS=/app/settings_filters/no_filters.json docker compose up
```

期间应观察到:rosetta healthy → gpu-worker 认领 job、设计循环启动(权重已在镜像内)→ relax/score 经 rosetta 执行 → 产物落到 `designs/PDL1_smoke/` → orchestrator 生成 `combined_final_stats.csv`。

结束与取产物:

```bash
docker compose down                      # 停掉常驻的 rosetta
docker run --rm -v bindcraft_workspace:/w alpine sh -c "ls -R /w/queue && tail -n 40 /w/queue/logs/*.log"
docker run --rm -v bindcraft_workspace:/w -v "$PWD/out":/out alpine cp -r /w/designs /out/
```

> 冒烟只验证链路连通,不代表能产出合格 binder(真实运行常需数百~数千条 trajectory)。

## 6. 正式批量运行

1. 在 `batch_inputs/` 放入若干 target JSON(schema 同 `settings_target/*.json`),`design_path` 指向 `/workspace/designs/<name>/`:
   ```json
   {
     "design_path": "/workspace/designs/PDL1/",
     "binder_name": "PDL1",
     "starting_pdb": "/app/example/PDL1.pdb",
     "chains": "A",
     "target_hotspot_residues": "56",
     "lengths": [65, 150],
     "number_of_final_designs": 100
   }
   ```
2. `docker compose up`(`.env` 已指向镜像站 tag,不会重新构建)。
3. 结束后 `docker compose down`,按上面方法导出 `designs/`。

## 7. 常见问题排查

| 现象 | 排查 |
|---|---|
| `python: executable file not found` | micromamba 镜像下,别用 `--entrypoint python`;命令直接作参数传(见 Level 1) |
| `gpu-worker` 卡在 "waiting for Rosetta service" | `docker compose logs rosetta`;确认 healthcheck 通过、`ROSETTA_URL=http://rosetta:8000` |
| 调 `/pr_relax` 服务断连 / 容器重启 | 确认服务是**单线程** `HTTPServer`(非 ThreadingHTTPServer),多线程会段错误崩溃 |
| RPC 报 `FileNotFound` | target `design_path` 不在 `/workspace` 下,或 rosetta 与 gpu-worker 未挂同一卷到同一路径 |
| 镜像拉取 `unknown blob` / EOF | 镜像站大层不稳,重试(已下载层会续传) |
| 多 worker 时 relax 很慢 | 提高 `ROSETTA_REPLICAS`(PyRosetta 单进程内串行,靠多副本并发) |
| GPU 设备列表为空 | 宿主装 NVIDIA 驱动 + nvidia-container-toolkit |
